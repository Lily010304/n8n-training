الجزء 1 · المفاهيم الأساسية

ما هي الأتمتة أصلًا؟

أي أتمتة في العالم — مهما كانت معقدة — تتكون من نفس الصيغة الثلاثية:

  • Trigger: «وصل بريد جديد»، «الساعة 7 صباحًا»، «زائر أرسل نموذجًا».
  • Logic: «هل البيانات صحيحة؟»، «هل هذا عميل جديد أم مكرر؟»، «أي قسم يخصّه هذا الطلب؟».
  • Action: «أرسل بريدًا»، «أضف صفًا في جدول»، «أنشئ مجلدًا»، «نبّه المدير».
الخلاصة

عندما تحلل أي مهمة تريد أتمتتها، اسأل دائمًا: ما الـ Trigger؟ ما القرارات في المنتصف؟ ما الأثر النهائي؟ — هذه العادة الذهنية أهم من أي أداة.

الجزء 1 · المفاهيم الأساسية

ما هو n8n؟

أداة أتمتة workflow automation مفتوحة المصدر تتيح ربط مئات الخدمات (Google، Slack، قواعد بيانات، أي API...) عبر واجهة رسم مرئية، مع إمكانية كتابة كود JavaScript عند الحاجة.

لماذا n8n وليس Zapier/Make؟

Self-hosting: تستضيفه على سيرفرك الخاص — بياناتك لا تغادر بنيتك التحتية، وهذا شرط أساسي لكثير من الشركات.

بدون حدود مصطنعة: لا تدفع لكل عملية تنفيذ، ولا تُقيَّد بعدد خطوات.

مرونة الكود: Code node كامل + HTTP Request لأي API حتى لو لم يكن له تكامل جاهز.

المصطلحات الأساسية

Workflow: المخطط الكامل — سلسلة العقد المتصلة التي تنفذ مهمة.

Node: خطوة واحدة في الـ workflow (استقبال، معالجة، إرسال...).

Execution: تشغيلة واحدة فعلية للـ workflow — لها سجل كامل يمكن فتحه ومعاينة بيانات كل عقدة فيها.

Canvas: مساحة الرسم التي تبني فيها الـ workflow بالسحب والإفلات.

الجزء 1 · المفاهيم الأساسية

تشريح الـ Workflow: كيف تتحرك البيانات بين العقد؟

كل عقدة تستقبل قائمة من العناصر (Items)، تعالجها، ثم تُخرج قائمة items جديدة للعقدة التالية. كل item هو كائن JSON واحد.

ما يخرج من عقدة «قراءة جدول الموظفين»
[ { "name": "Sara", "dept": "Design" }, { "name": "Omar", "dept": "IT" }, { "name": "Lina", "dept": "HR" } ]
العقدة التالية «إرسال بريد» تعمل 3 مرات
Item 1 → email to Sara Item 2 → email to Omar Item 3 → email to Lina
القاعدة الذهبية

أغلب العقد تنفَّذ مرة لكل item تلقائيًا — لا تحتاج حلقة تكرار يدوية. إذا وصلها 100 عنصر، سترسل 100 بريدًا. هذا الفهم يمنع أكثر الأخطاء شيوعًا عند المبتدئين.

الجزء 2 · أساسيات خارج n8n

كيف تتحرك البيانات على الإنترنت؟

قبل أي أتمتة، يجب فهم النموذج الأساسي للاتصال: Request / Response.

  • الـ Client يسأل دائمًا أولًا — السيرفر لا يبادر بالإرسال (إلا في حالة الـ Webhooks، وسنصل لها).
  • كل طلب يذهب إلى عنوان URL محدد، وكل استجابة تعود مع Status Code يلخّص النتيجة.
  • البيانات المتبادلة بين الأنظمة اليوم تكون غالبًا بصيغة JSON.
الجزء 2 · أساسيات خارج n8n

ما هو الـ API؟

الـ API (Application Programming Interface) هو «باب رسمي» تفتحه خدمة ما ليتعامل معها الكود بدل البشر. الموقع واجهة للإنسان، والـ API واجهة للبرامج — نفس البيانات، بابان مختلفان.

HTTP Method المعنى مثال واقعي
GET اقرأ بيانات — بدون أي تعديل جلب حالة الطقس، قراءة قائمة العملاء
POST أنشئ شيئًا جديدًا إضافة عميل جديد، إرسال رسالة
PUT / PATCH عدّل شيئًا موجودًا تحديث بريد موظف، تغيير حالة طلب
DELETE احذف حذف سجل، إلغاء حجز
لماذا يهم في n8n؟

كل عقدة تكامل في n8n (Gmail، Sheets، Slack...) هي مجرد غلاف جميل حول API. وعندما لا يوجد تكامل جاهز لخدمة ما، عقدة HTTP Request تتيح مخاطبة أي API في العالم مباشرة — لذلك من يفهم الـ APIs لا يقف عند حدود قائمة التكاملات أبدًا.

الجزء 2 · أساسيات خارج n8n

JSON — لغة البيانات التي يتكلمها كل شيء

JSON (JavaScript Object Notation) هو تنسيق نصي بسيط لتمثيل البيانات. كل ما يمر عبر n8n — مدخلات، مخرجات، ردود APIs — هو JSON.

البنية الأساسية
{ "name": "Sara", // نص (string) "age": 28, // رقم (number) "active": true, // منطقي (boolean) "skills": ["n8n", "SQL"] // مصفوفة (array) }
التداخل (nesting) + الوصول للقيم
{ "employee": { "name": "Sara", "manager": { "email": "ali@corp.com" } } } // للوصول لبريد المدير: {{ $json.employee.manager.email }}
مهارة عملية أساسية

اقرأ أي JSON من الخارج للداخل: كائن ← داخله حقول ← بعض الحقول كائنات أو مصفوفات بدورها. القدرة على النظر إلى استجابة API ضخمة واستخراج «مسار» القيمة المطلوبة هي المهارة الأكثر استخدامًا يوميًا في n8n.

Status Codes والمصادقة (Authentication)

أكواد الحالة التي ستقابلها يوميًا

الكودالمعنىماذا تفعل؟
200 / 201نجاح / تم الإنشاءأكمل طبيعيًا
400طلبك أنت خاطئ (بيانات ناقصة/غير صالحة)صحّح البيانات — الإعادة لن تنفع
401 / 403مشكلة مصادقة / صلاحياتافحص الـ credential — لا Retry
429تجاوزت حد الطلبات (rate limit)Retry مع انتظار أطول
500 / 503السيرفر نفسه متعطل مؤقتًاRetry with backoff

طرق المصادقة الشائعة

  • API Key: مفتاح نصي ثابت يُرسل مع كل طلب (في header أو query) — الأبسط والأكثر شيوعًا.
  • Bearer Token: رمز مؤقت في header بصيغة Authorization: Bearer <token>.
  • OAuth2: المستخدم يوافق عبر شاشة رسمية (مثل «Sign in with Google») والنظام يحصل على token محدود الصلاحيات قابل للتجديد — هذا ما تستخدمه عقد Google في n8n.
الجزء 2 · أساسيات خارج n8n

Webhook مقابل Polling — كيف نعرف أن شيئًا حدث؟

Polling — «اسأل كل فترة»

n8n يسأل الخدمة بشكل دوري: «هل من جديد؟» عبر Schedule Trigger + طلب GET.

مثال

كل 15 دقيقة: افحص إن كان هناك صف جديد في الجدول.

العيب: تأخير حتى الفحص التالي + طلبات كثيرة بلا فائدة أغلب الوقت.

Webhook — «أخبرني فور الحدوث»

أنت تعطي الخدمة رابط URL خاص بك (يولّده Webhook node)، وهي تستدعيه فورًا عند وقوع الحدث.

مثال

نموذج تسويقي يُرسل بيانات كل Lead جديد للحظة تسجيله.

الميزة: فوري وفعّال — وهو أساس أغلب أنظمة الإنتاج.

تنبيه Production مهم

الخدمات المرسلة تُعيد إرسال الـ Webhook عند أي تأخر في الرد (retry behavior) — لذلك أي workflow يبدأ بـ Webhook يجب أن يتحمل استقبال نفس الطلب مرتين. هذا بالضبط سبب حاجتنا لمفهوم Idempotency القادم في الجزء الخامس.

الجزء 3 · أنواع الـ Nodes

خريطة أنواع العقد في n8n

مئات العقد في n8n تنتمي كلها إلى 5 عائلات فقط. من يفهم العائلات يستطيع بناء أي شيء:

  • Triggers تبدأ التنفيذ: Manual، Schedule، Webhook، Form، وTriggers التطبيقات (بريد جديد، صف جديد...).
  • Actions تنفّذ أثرًا في خدمة خارجية: Gmail، Google Sheets، Slack، HTTP Request...
  • Logic تتحكم بمسار التنفيذ: IF، Switch، Filter، Merge، Wait، Loop.
  • Data تعيد تشكيل البيانات نفسها: Set/Edit Fields، Code، Aggregate، Split Out، Remove Duplicates.
  • AI عقد الذكاء الاصطناعي: AI Agent، LLM Chain، Embeddings، Vector Stores، Memory.

عائلة الـ Actions — حيث يقع الأثر الفعلي

عقد التكاملات الجاهزة تغطي مئات الخدمات، وكل واحدة تعرض Operations متعددة (أرسل، اقرأ، حدّث، احذف...):

  • Gmail / Send Email: إرسال بريد — العقدة التي ستستخدمها كل مشاريعنا الأربعة.
  • Google Sheets: Append / Get / Update rows — أبسط «قاعدة بيانات» للبدء وأداة التتبع الأشهر.
  • Google Drive / Docs / Calendar: ملفات ومستندات وأحداث — عماد نظام HR Onboarding.
  • Slack / Telegram / WhatsApp: إشعارات فورية للفرق.
العقدة الأهم على الإطلاق: HTTP Request

عندما لا تجد تكاملًا جاهزًا لخدمة ما، عقدة HTTP Request تخاطب أي API مباشرة: تحدد الـ URL والـ Method والـ Headers والـ Body بنفسك. إتقانها = لا حدود لما يمكن ربطه. وهي أيضًا بوابة الـ Scraping التي سنراها في الجزء السادس.

عائلة الـ Logic — التحكم بمسار التنفيذ

العقدةماذا تفعل؟مثال
IF تفرّع ثنائي: true / false هل البريد صالح؟ نعم → أكمل، لا → ارفض برسالة
Switch تفرّع متعدد المسارات وجّه الطلب حسب القسم: HR / IT / Finance
Filter يُبقي items تحقق شرطًا ويُسقط الباقي أبقِ فقط العقارات تحت 100 ألف
Merge يجمع مخرجات فرعين في مسار واحد دمج نتائج Doc والـ Calendar بعد تنفيذهما بالتوازي
Wait يُجمّد التنفيذ حتى وقت أو حدث انتظار موافقة المدير (Approval Gate)
Loop Over Items معالجة على دفعات محكومة معالجة 500 سجل بدفعات من 50 لتفادي rate limits
الفرق بين IF و Filter — سؤال شائع

‏IF يقسم المسار لفرعين وكلاهما قد يُكمل بمنطق مختلف. Filter لا يتفرع — فقط يُسقط العناصر غير المطابقة ويكمل بمسار واحد.

العقدةماذا تفعل؟مثال

الجزء 4 · AI Agents

قبل الـ Agent: ما الذي يفعله نموذج اللغة (LLM) فعلًا؟

الـ LLM (مثل GPT أو Claude أو Gemini) يستقبل نصًا (Prompt) ويولّد نصًا إكمالًا له. لا يعرف بياناتك الخاصة، ولا «يتذكر» شيئًا بين الطلبات ما لم تُرسل له السياق بنفسك.

  • Prompt: التعليمات + البيانات التي ترسلها للنموذج. جودة المخرجات = جودة الـ prompt أولًا.
  • Tokens: وحدة قياس النص (كلمة ≈ 1–2 token). التكلفة والحدود تُحسب بها.
  • Temperature: درجة العشوائية — منخفضة للتصنيف والاستخراج، أعلى للكتابة الإبداعية.
القاعدة الأهم للإنتاج

مخرجات الـ LLM غير مضمونة الشكل: قد يضيف شرحًا لم تطلبه أو يكسر تنسيق JSON. لذلك في أي أتمتة جادة نطلب منه صيغة JSON محددة ثم نتحقق منها (validate) قبل تمريرها لأي خطوة ذات أثر — لا نثق بالمخرجات ثقة عمياء أبدًا.

الجزء 4 · AI Agents

عقدة AI Agent: نموذج + أدوات + ذاكرة

الفرق بين «نداء LLM عادي» و«Agent»: الـ Agent يستطيع اتخاذ قرار باستخدام أداة، قراءة نتيجتها، ثم تقرير الخطوة التالية — في حلقة ذكية.

مثال: عميل يسأل «أين طلبي رقم 4512؟» → الـ Agent يقرر استخدام أداة «بحث في جدول الطلبات» → يقرأ النتيجة → يصيغ ردًا بشريًا مخصصًا. بدون أدوات، كان سيؤلف جوابًا من خياله.

الجزء 4 · AI Agents

RAG — كيف يجيب الـ AI من مستنداتك أنت؟

RAG (Retrieval-Augmented Generation): بدل أن يجيب النموذج من معرفته العامة، نُحضر له المقاطع الأكثر صلة من مستنداتنا ونطلب الإجابة منها فقط.

لماذا يهم في الـ Production؟

‏RAG يقلل «الهلوسة» ويجعل الإجابات قابلة للتوثيق (من أي مستند جاءت؟). سنبنيه فعليًا في نظام مساعد الدعم الفني (الجلسات 6–7) على مستندات حقيقية.

الجزء 4 · AI Agents

ضوابط أمان الـ AI في الأتمتة

  1. Structured Output + تحقق اطلب من النموذج JSON بمخطط (schema) محدد، ثم افحص الناتج في عقدة تالية: هل الحقول موجودة؟ القيم ضمن المسموح؟ أي مخالفة → مسار fallback، لا متابعة عمياء.
  2. Human-in-the-loop للقرارات الحساسة أي فعل لا يمكن التراجع عنه (إرسال لعميل، حذف، دفع) يمر ببوابة موافقة بشرية (Wait node / Approval Gate) — الـ AI يقترح، الإنسان يعتمد.
  3. خطة Fallback دائمًا ماذا لو تعطل مزود الـ LLM أو تجاوز الرد المهلة؟ حدد مسارًا بديلًا: رد افتراضي آمن، تحويل لموظف بشري، أو إعادة محاولة بنموذج آخر.
  4. حدود التكلفة والاستخدام راقب استهلاك الـ tokens وضع حدودًا — حلقة agent خارجة عن السيطرة قد تكلف كثيرًا في ساعات.
الجزء 5 · جاهزية الإنتاج

أعمدة الـ Production الأربعة

هذه المفاهيم شُرحت بعمق في الجلسة الأولى — هنا نضعها كخريطة واحدة متكاملة:

1 · Error Handling

Error Trigger Workflow يلتقط أي فشل غير متوقع وينبّه الفريق. Continue On Fail يمنع عنصرًا فاشلًا واحدًا من إيقاف الدفعة. Retry with Backoff للأخطاء المؤقتة (503، 429) فقط — لا فائدة منه مع 400 و401.

2 · Idempotency

الـ workflow يجب أن يتحمل التشغيل مرتين بنفس المدخلات دون تكرار الآثار. النمط: ابحث قبل أن تُنشئ — افحص سجل التتبع أولًا، وإن وُجد السجل فتخطَّ أو حدّث.

3 · المراقبة والسجلات

سجل تتبع (Sheets/DB) يوثق كل عملية وحالتها، تنبيهات فورية عند الفشل، وسجلات تدقيق (Audit Logs) تجيب لاحقًا: ماذا حدث؟ متى؟ ولماذا؟ — لا صمت أبدًا.

4 · أمان الـ Credentials

المفاتيح في مخزن n8n المشفّر فقط — أبدًا داخل Code node أو Sticky Note. أضيق صلاحيات ممكنة (least privilege): صلاحية إرسال بريد فقط لا تحتاج قراءة صندوق الوارد كاملًا.

الجزء 5 · جاهزية الإنتاج

عادات تصميم تفصل المحترف عن الهاوي

  1. Validation دائمًا قبل أي أثر جانبي تحقق من البيانات فور استقبالها — قبل أي إنشاء أو إرسال. بيانات خاطئة تدخلت متأخرًا = آثار جزئية يصعب التراجع عنها.
  2. رد واضح لكل مسار نجاح، رفض validation، تكرار، فشل — كل مسار ينتهي برد JSON أو رسالة واضحة. الفشل الصامت هو أسوأ أنواع الفشل.
  3. قسّم الأنظمة الكبيرة إلى Sub-workflows ‏workflow واحد عملاق يصعب اختباره وصيانته. قسّمه لوحدات لكل منها مدخلات ومخرجات واضحة (contract) — مثل الدوال في البرمجة.
  4. إعدادات مركزية في عقدة Config كل الـ IDs والقيم القابلة للتغيير في عقدة Set واحدة أول الـ workflow — تغيير بيئة العمل يصبح تعديل مكان واحد.
  5. اختبر مسارات الفشل قبل مسار النجاح جرّب البيانات الناقصة والمكررة والـ API المتعطل — النجاح السعيد يختبر نفسه بنفسه.
الجزء 6 · الأتمتة حسب المجال

نفس الأنماط، مجالات مختلفة — 1

HR — الموارد البشرية

حالات: Onboarding موظفين (نظامنا الرئيسي)، طلبات إجازات مع موافقات، تذكيرات بانتهاء العقود وفترات التجربة.

الأنماط الحاكمة: Approval Gates (موافقة المدير)، Idempotency (لا حسابات مكررة)، تتبع الحالة لكل موظف في سجل مركزي.

Marketing — التسويق

حالات: استقبال Leads من النماذج الإعلانية → إثراء بياناتهم (تكبير الصورة عنهم من مصادر أخرى) → تصنيفهم عبر LLM → رسالة أولى مخصصة لكل شريحة.

الأنماط الحاكمة: Webhook انتقال فوري، Deduplication (نفس الشخص من حملتين)، تصنيف LLM بمخرجات JSON مُتحقق منها، ثم Routing حسب الشريحة.

لاحظ النمط المتكرر

استقبال → تحقق → إزالة تكرار → إثراء/تصنيف → فعل → تسجيل وتنبيه. تتغير الخدمات والأسماء، لكن الهيكل واحد في كل مجال — هذا جوهر مبدأ «إعادة استخدام الأنماط بين الأنظمة».

الجزء 6 · الأتمتة حسب المجال

نفس الأنماط، مجالات مختلفة — 2

Real Estate — العقارات

حالات: تجميع عقارات جديدة من مصادر متعددة (Scraping/APIs)، مطابقتها آليًا مع معايير كل عميل (سعر، منطقة، مساحة)، تنبيه فوري عند فرصة مطابقة، وحجز معاينة بموافقة.

الأنماط الحاكمة: Scheduled scraping + Dedup، منطق مطابقة (Filter/Code)، تنبيهات فورية، Approval Gate للحجز.

Email — البريد كقناة تنفيذ

حالات: بريد ترحيب مخصص، نشرات مجمعة (digests)، تنبيهات فشل للفريق التقني، وملخصات دورية للإدارة.

الأنماط الحاكمة: بناء النص عبر Expressions من بيانات حية، تجميع قبل الإرسال (بريد واحد مجمّع لا 20 بريدًا)، وRetry على عقدة الإرسال لأن خدمات البريد تتعثر مؤقتًا.

الـ AI/Support (مساعد الدعم بـ Agent + RAG) رأيناه في الجزء الرابع — وهو خامس مجالات الدورة الرئيسية.

الجزء 6 · الأتمتة حسب المجال

‏Scraping البيانات وتنظيفها — الـ Pipeline الكامل

الـ Scraping هو جلب بيانات من مصادر لم تُصمَّم لتعطيك إياها بشكل مرتب. الخط الكامل:

  • Fetch: الأفضل دائمًا API رسمي أو RSS إن وُجد؛ الـ HTML scraping آخر الحلول (هشّ ويتكسر مع أي تغيير تصميم).
  • Parse: عقدة HTML Extract (بمحددات CSS) أو Code node لاستخراج الحقول من النص الخام.
  • Clean: توحيد التنسيقات — «1,000 دينار» و«1000 JOD» يجب أن يصبحا نفس الرقم؛ إزالة مسافات ورموز زائدة؛ حقول فارغة → قيم افتراضية.
  • Dedup: Remove Duplicates بمعرّف ثابت (URL، رقم إعلان) + مقارنة مع ما خُزّن سابقًا.
  • Store + Act: التخزين أولًا، ثم الفعل (تقرير، تنبيه، مطابقة) — حتى لا يضيع ما جُمع إن فشل الفعل.
أخلاقيات وقانون

احترم شروط استخدام المواقع وملف robots.txt، لا تُغرق المصدر بالطلبات (فواصل زمنية بين الطلبات)، ولا تجمع بيانات شخصية دون أساس قانوني.

4 مشاريع صغيرة — كلها تُرسل بريدًا، وكلها تبني نحو الأنظمة الكبيرة

لماذا هذا الترتيب؟

كل مشروع يضيف طبقة واحدة جديدة فوق سابقه: جلب بيانات → تحقق → ذاكرة ومنع تكرار → نظام مصغّر متكامل. المشروع الرابع هو «بروفة» مصغّرة لنظام HR Onboarding الكامل.

مشروع 1 · سهل

نشرة الطقس الصباحية — Daily Weather Digest

الهدف: كل صباح 7:00، اجلب طقس اليوم من Open-Meteo API (مجاني، بدون مفتاح)، وأرسل بريدًا ملخصًا — مع تحذير مظلة إن كان احتمال المطر مرتفعًا.

نقاط البناء المهمة

  • الـ HTTP Request يمرر خط الطول/العرض كـ Query Parameters ويطلب daily=temperature_2m_max,precipitation_probability_max.
  • عقدة Set تبني الرسالة من $json، مع Ternary للتحذير الشرطي:
{{ $json.daily.precipitation_probability_max[0] > 40 ? 'خذ مظلة معك ☔' : 'يوم صافٍ غالبًا ☀️' }}
  • اختبر يدويًا (Execute Workflow) قبل تفعيل الجدولة — عادة احترافية دائمة.
مشروع 2 · سهل

إشعار نموذج تواصل — Contact Form Notifier

الهدف: نموذج تواصل (اسم، بريد، رسالة) → تحقق من البيانات → بريد فوري لصاحب الموقع، أو رسالة رفض واضحة تحدد الحقل المشكلة.

الدرس المحوري

لاحظ مكان Send Email: بعد الـ Validation في مسار true فقط. التحقق قبل أي أثر جانبي — نفس القاعدة الذهبية من عرض الجلسة الأولى، بأصغر مثال ممكن.

مشروع 3 · أصعب قليلًا

نشرة RSS بدون تكرار — RSS Digest with Deduplication

الهدف: يوميًا: اقرأ مقالات من RSS، استبعد ما أُرسل سابقًا (Google Sheet كذاكرة)، أرسل بريدًا واحدًا مجمّعًا بالجديد فقط، ثم سجّل ما أرسلته.

  • لماذا Code node للفلترة؟ نقارن روابط المقالات الجديدة بقائمة الروابط المخزنة — مقارنة قائمتين تحتاج منطقًا يتجاوز الـ Filter البسيط.
  • لماذا Aggregate قبل الإرسال؟ بريد واحد منظم أفضل من 15 بريدًا متتاليًا — نفس منطق الإشعارات المجمّعة في الأنظمة الكبيرة.
  • ترتيب مقصود: الإرسال ثم التسجيل — إن فشل الإرسال لا نسجل المقالات كمُرسلة، فتُعاد المحاولة غدًا تلقائيًا.
مشروع 4 · أصعب قليلًا

استقبال Leads مع فحص التكرار — Lead Capture

الهدف: ‏Webhook يستقبل Lead (اسم، بريد، شركة) → تحقق → فحص تكرار في جدول العملاء → تسجيل + بريد ترحيب مخصص (بـ Retry) → رد JSON واضح لكل مسار.

  • Retry على Send Email فقط: خدمة البريد قد تتعثر مؤقتًا (خطأ transient) — 3 محاولات بفاصل متضاعف. لكن لا Retry على الـ Validation: البيانات الخاطئة ستبقى خاطئة.
  • Optional Chaining في نص الرسالة: {{ $json.company?.name }} — الشركة حقل اختياري ولا يجوز أن يكسر الإرسال.
  • اختبار إلزامي بـ 3 حالات: بيانات صالحة، بيانات ناقصة، ونفس البريد مرتين — الثالثة تثبت أن الـ Idempotency يعمل فعلًا.
