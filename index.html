الجزء الأول · النظرية

"Workflows that work" مقابل "Systems that survive"

الفرق الحقيقي ليس في قدرة الـ Workflow على النجاح في أول تشغيل، بل في قدرته على الصمود عند تكرار التشغيل، أو فشل API خارجي، أو وصول بيانات غير متوقعة.

لماذا يهم في الـ Production؟

لأن أي Workflow حقيقي سيُشغَّل آلاف المرات، وسيواجه حتمًا Webhook مكررًا، أو API بطيئًا، أو حقلًا فارغًا — والفرق بين النوعين لا يظهر إلا تحت هذا الضغط.

الجزء الأول · النظرية

مثال ملموس

Workflow لإرسال بريد ترحيبي يعمل تمامًا في أول اختبار. لكن الخدمة الخارجية التي تُشغّل الـ Webhook تُعيد إرساله تلقائيًا عند أي تأخير في الرد (retry behavior افتراضي)، فيُشغَّل نفس الـ Workflow مرتين، ثم عشرين مرة، حتى تصل النتيجة إلى إرسال 200 بريد إلكتروني مكرر لنفس الموظف.

الخلاصة

الـ Workflow نجح تقنيًا في كل مرة — لكن النظام فشل. هذا بالضبط الفرق بين workflow that works وsystem that survives.

الجزء الأول · Expressions المتقدمة

التعبيرات المتقدمة في n8n

الـ Expressions هي الجسر بين الـ Nodes، وإتقانها يحدد مدى مرونة الـ Workflow بالكامل. نبدأ بأبسطها وأكثرها استخدامًا.

$json — بيانات العنصر الحالي

يشير $json إلى بيانات الـ item الحالي كما وصلت إلى هذه الـ Node تحديدًا.

قبل (البيانات الواردة)
{ "name": "Sara", "email": "sara@example.com" }
بعد (الاستخدام)
{{ $json.email }}
→ sara@example.com
الجزء الأول · Expressions المتقدمة

الوصول لعقدة أخرى وحساب عدد العناصر

$node["NodeName"].json

يسمح بجلب بيانات عقدة سابقة محددة بالاسم، وليس فقط العقدة التي تسبق الحالية مباشرة.

قبل

عقدة باسم Get Manager أرجعت:

{ "managerEmail": "ali@corp.com" }
بعد
{{ $node["Get Manager"].json.managerEmail }}
→ ali@corp.com

$items()

يُرجع كل الـ items التي وصلت للعقدة الحالية — مفيد لمعرفة العدد الكلي أو المعالجة الجماعية.

قبل

5 عناصر (موظفين) وصلت للعقدة الحالية

بعد
{{ $items().length }}
→ 5
الجزء الأول · Expressions المتقدمة

$now وحساب التواريخ عبر Luxon

‏$now هو كائن Luxon DateTime يمثل الوقت الحالي، ويدعم دوال حساب التواريخ مباشرة (جمع، طرح، تنسيق).

قبل

نريد تاريخ تجهيز الحساب قبل تاريخ الالتحاق بثلاثة أيام

بعد
{{ $now.plus({ days: 3 }).toFormat('yyyy-MM-dd') }}
لماذا يهم في الـ Production؟

بدون Luxon ستحتاج لكتابة منطق تواريخ يدوي عرضة للأخطاء، خصوصًا مع فروقات التوقيت (timezones) — موضوع سنعود له بالتفصيل في الجلسة القادمة.

الجزء الأول · Expressions المتقدمة

Ternary وOptional Chaining

Ternary

تعبير شرطي مختصر بصيغة condition ? valueIfTrue : valueIfFalse بدل كتابة جملة if كاملة.

قبل

‏$json.department قد تكون فارغة

بعد
{{ $json.department ? $json.department : 'غير محدد' }}

Optional Chaining (?.)

يمنع توقف الـ Workflow بخطأ عندما يكون أحد الحقول الوسيطة غير موجود (null/undefined).

قبل (بدون ?.)
{{ $json.manager.email }}
→ خطأ إذا manager غير موجودة
بعد (مع ?.)
{{ $json.manager?.email }}
→ undefined بدل توقف التنفيذ
الجزء الأول · Error Handling

الـ Error Handling كجزء من التصميم، وليس ترقيعًا لاحقًا

في نظام Production نحتاج لأكثر من آلية واحدة، وكل آلية تُستخدم في موقف مختلف تمامًا.

Error Trigger Workflow

Workflow منفصل يحتوي عقدة Error Trigger، يُربط كـ Error Workflow لأي workflow آخر، فيلتقط أي فشل غير متوقع تلقائيًا.

مثال

‏workflow الـ onboarding يفشل بسبب انقطاع اتصال بـ Google API ← الـ Error Trigger workflow يُشغَّل تلقائيًا ويرسل تنبيهًا لفريق الـ IT.

Continue On Fail

إعداد على مستوى الـ Node الواحدة يجعل تنفيذ الـ workflow يكمل حتى لو فشلت هذه العقدة تحديدًا، بدل توقف كل شيء.

مثال

عقدة إرسال بريد الترحيب تفشل لموظف واحد بسبب بريد خاطئ ← باقي الموظفين في نفس التشغيل الجماعي يكملون بشكل طبيعي.

الجزء الأول · Error Handling

Retry with Backoff على عقد HTTP Request

إعادة محاولة الطلب تلقائيًا عند الفشل، مع فاصل زمني متزايد بين كل محاولة، بدل إعادتها فورًا وبشكل متكرر.

مثال: عقدة HTTP Request تنشئ حساب بريد للموظف عبر API خارجي، مضبوطة على 3 محاولات بفاصل يبدأ من 2 ثانية ويتضاعف في كل مرة. إذا كان الـ API بطيئًا مؤقتًا، تنجح المحاولة الثانية أو الثالثة بدل فشل الـ workflow بالكامل.

لماذا يهم في الـ Production؟

بدون backoff، إعادة المحاولة الفورية والمتكررة قد تُحمّل الـ API المتعثر أصلًا بضغط إضافي، وأحيانًا تُفعّل rate limiting وتزيد الوضع سوءًا.

الجزء الأول · Error Handling

متى Retry؟ ومتى لا؟ — جدول القرار

الموقف الإجراء الصحيح السبب
API يرجع 503 Service Unavailable Retry with backoff خطأ مؤقت (transient) — النظام الخارجي سيتعافى غالبًا خلال ثوانٍ.
بيانات إدخال غير صالحة (invalid input) بدون Retry — توجيه لمراجعة الخطأ منطقي وسيتكرر مهما أعدنا المحاولة؛ يحتاج تصحيح بيانات أو تدخل بشري.
API يرجع 429 Too Many Requests Retry with backoff (فاصل أطول) تجاوزنا rate limit مؤقتًا، والانتظار يحل المشكلة تلقائيًا.
خطأ Authentication / 401 بدون Retry — تنبيه فوري إعادة المحاولة لن تُصلح credential منتهي أو خاطئ؛ يحتاج تدخل فوري.
الجزء الأول · Idempotency

لماذا يجب أن يتحمّل الـ Workflow التشغيل مرتين؟

عملية Idempotent هي عملية تُنتج نفس النتيجة النهائية سواء نُفذت مرة واحدة أو عدة مرات بنفس المدخلات.

لماذا يهم في الـ Production؟

في n8n، الـ Webhook قد يُعاد إرساله من الخدمة المرسلة، والمستخدم قد يضغط submit مرتين، وأنت شخصيًا قد تُعيد تشغيل execution فاشلة يدويًا — وفي كل هذه الحالات، بدون Idempotency ستتكرر الآثار الجانبية (duplicate side effects).

الجزء الأول · Idempotency

مثال: نظام HR Onboarding

قبل إنشاء أي حساب، نتحقق أولًا: هل يوجد بالفعل سجل بنفس الـ email في جدول الـ Google Sheets الذي نتتبع فيه حالة كل موظف؟

✕
بدون فحص: كل تشغيل — حتى المكرر — يؤدي لإنشاء حساب بريد جديد + Drive folder جديد.
✓
مع فحص Idempotency: تشغيل مكرر لنفس الموظف ← البحث في Sheets يجد سجلًا موجودًا ← نتخطى خطوات الإنشاء أو نُحدّث السجل الحالي فقط.
الجزء الأول · Credentials Hygiene

Credentials Hygiene

إدارة الـ API keys والـ tokens بطريقة آمنة: تخزين مشفّر، صلاحيات محدودة (least-privilege scopes)، وعدم كتابتها داخل الكود مباشرة.

لماذا يهم في الـ Production؟

أي Code node أو Sticky Note يحتوي على token مكشوف يتحول لثغرة أمنية فور مشاركة الـ workflow أو رفعه لأي مكان — والصلاحيات الواسعة تعني أن أي اختراق صغير قد يتحول لكارثة كبيرة.

الجزء الأول · Credentials Hygiene

الطريقة الخاطئة مقابل الصحيحة

✕
كتابة الـ API token كنص مباشر داخل Code node:
const token = "sk-live-abc123...";
fetch(url, { headers: { Authorization: token } });
✓
حفظ الـ token داخل الـ Credentials المشفّرة في n8n (Credentials store)، وربطه بالـ node عبر الواجهة، مع تفعيل أضيق الصلاحيات الممكنة فقط — مثلًا: Google OAuth Scope لإنشاء ملفات في Drive فقط، وليس صلاحية كاملة على الحساب.
الجزء الثاني · البناء

نظرة عامة: نظام HR Onboarding Engine

نبني اليوم الجزء الأول من نظام أكبر — من الـ intake حتى الـ Notifications. خطوة الـ Approval (بالحدود المتقطعة) ستُضاف في الجلسة القادمة.

الحدود المتقطعة = خطوة تُبنى في الجلسة القادمة

الجزء الثاني · خطوة البناء 1

إعداد الـ Intake

نبدأ بـ Form Trigger (أو Webhook إذا كان الإدخال من نظام خارجي مثل ATS). الحقول المطلوبة في هذه المرحلة:

  • الاسم الكامل للموظف
  • البريد الشخصي (personal email)
  • القسم (department)
  • تاريخ الالتحاق (start date)
  • بريد المدير المباشر (manager email)
لماذا Form وليس Webhook هنا؟

‏Form Trigger أنسب لأن فريق HR سيُدخل البيانات يدويًا عبر واجهة بسيطة، بينما الـ Webhook يُفضَّل عند التكامل الآلي مع نظام خارجي مثل ATS.

الجزء الثاني · خطوة البناء 2

منطق الـ Validation

  • الحقول الإجبارية (required fields): رفض أي إدخال ينقصه اسم، email، أو start date.
  • تنسيق الـ email: فحص عبر regex بسيط، ورفض أي قيمة لا تطابق نمط بريد صالح.
  • منطقية تاريخ البدء (start-date sanity): رفض تاريخ في الماضي، أو تاريخ بعيد جدًا في المستقبل (مثلًا أكثر من 6 أشهر).
  • عند الرفض: لا نُسقط البيانات بصمت — نُعيد رسالة واضحة عبر Form/Webhook response توضح تحديدًا أي حقل غير صالح ولماذا.
{{ $json.startDate ? ($json.startDate < $now.toISODate() ? 'invalid: past date' : 'ok') : 'invalid: missing' }}
الجزء الثاني · خطوة البناء 3

ترتيب الفروع: Sequential أم Parallel؟

ليست كل خطوات الـ Provisioning متساوية في الأولوية — بعضها يعتمد على نتيجة خطوة أخرى، وبعضها مستقل تمامًا.

  1. Sequential — حساب البريد أولًا إنشاء حساب البريد (email account) يجب أن يكتمل أولًا، لأن خطوة منح صلاحيات الوصول لمجلد الـ Drive (folder permissions) تحتاج إلى الـ email الجديد كمُدخل.
  2. Parallel — Doc والـ Calendar معًا توليد وثيقة الترحيب (welcome doc) وإنشاء أحداث التقويم (calendar events) لا يعتمدان على بعضهما، فيمكن تشغيلهما في فرعين متوازيين لتقليل الزمن الكلي للتنفيذ.
الجزء الثاني · البناء

أخطاء شائعة عند بناء أنظمة كهذه

1

Silent failures

عقدة تفشل بصمت دون Error Trigger أو أي تنبيه، فلا يكتشف أحد المشكلة إلا بعد شكوى موظف لم يستلم حسابه.

2

Validation متأخرة

التحقق من صحة البيانات بعد أن بدأت بالفعل خطوات إنشاء الحسابات، مما يترك آثارًا جانبية جزئية يصعب التراجع عنها.

3

Workflow واحد عملاق

بناء كل شيء — من الـ intake حتى الـ notifications — داخل workflow واحد ضخم يصعب قراءته أو تصحيحه أو اختباره بمعزل عن الباقي.

الجزء الثالث · الختام

الواجب

  1. أكمل الفروع غير المنتهية في نظام HR Onboarding، خصوصًا فرع الـ Drive folder وفرع الـ welcome doc.
  2. اكتب قائمة سيناريوهات الفشل الواقعية لكل حالة فشل ممكنة لهذا النظام، مثل: انقطاع الـ API، تكرار إرسال نفس الـ submission، أو بريد مدير (manager email) غير صحيح.
  3. صنّف كل سيناريو لكل سيناريو فشل كتبته، حدد: هل يحتاج Retry؟ أم Continue On Fail؟ أم رفض فوري في مرحلة الـ Validation؟
الجزء الثالث · الختام

مواضيع الجلسة القادمة

  • Human-in-the-loop approvals عبر Wait node.
  • تتبع حالة كل طلب كـ state machine داخل Sheets أو قاعدة بيانات.
  • Sub-workflows بعقود input/output واضحة (input/output contracts).
  • التعامل مع فروقات التوقيت (timezone handling).
  • ربط الـ global error handler فعليًا بكل الـ workflows في النظام.
اختبار سريع السؤال 1 من 6

ما الفرق الصحيح بين $json و $node["NodeName"].json؟

‏$json = بيانات الـ item الحالي في السياق الحالي. $node["NodeName"].json = سحب بيانات من عقدة سابقة محددة بالاسم — مفيد عند الحاجة لمعلومة من عقدة أبعد من التي تسبق الحالية مباشرة.
اختبار سريع السؤال 2 من 6

كيف نحصل على تاريخ بعد 3 أيام من الآن باستخدام $now؟

‏$now هو كائن Luxon DateTime، لذلك حساب التواريخ يتم عبر دوال Luxon مثل .plus({ days: 3 }) وليس عمليات جمع عادية على أرقام.
اختبار سريع السؤال 3 من 6

ما فائدة استخدام ?. في تعبير مثل $json.manager?.email؟

‏Optional chaining يمنع خطأ التنفيذ عندما تكون إحدى الحلقات الوسيطة في المسار مفقودة أو null، ويعيد undefined بدل إيقاف الـ execution بالكامل.
اختبار سريع السؤال 4 من 6

لماذا يجب أن يكون Workflow إنشاء حسابات الموظفين Idempotent؟

التشغيل المكرر (retries، إعادة تنفيذ يدوية، إعادة إرسال webhook) وارد جدًا في الإنتاج؛ فحص وجود السجل قبل الإنشاء يمنع الآثار الجانبية المكررة.
اختبار سريع السؤال 5 من 6

الـ API يرجع 503 Service Unavailable بشكل مؤقت، ما الإجراء الصحيح؟

‏503 غالبًا خطأ مؤقت (transient) — الاستجابة الصحيحة هي Retry with backoff، بعكس الأخطاء المنطقية/الدائمة التي لا تستفيد من إعادة المحاولة.
اختبار سريع السؤال 6 من 6

أين يجب أن تتم عملية Validation في نظام HR Onboarding؟

الـ Validation يجب أن تتم مباشرة بعد الـ intake، قبل أي خطوة Provisioning ذات أثر جانبي — هذا يمنع إنشاء حسابات أو ملفات جزئية بناءً على بيانات خاطئة.
